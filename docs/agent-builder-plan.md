## 1. Elasticsearch Indices

All indices are created by `npm run setup:elastic` (idempotent).

### `granola-meeting-notes` (existing, extended)

The primary data store. Every enriched meeting note lives here.

Key new fields added for the agent pipeline:
- `summary_embedding` — dense_vector via `.jina-embeddings-v3` (Elastic-managed EIS)
- `transcript_semantic` — dense_vector via `.jina-embeddings-v3`

The ingest pipeline (`granola-notes-pipeline`) runs automatically on every index operation.

### `account-pursuit-team`

One document per account. Document `_id` = account name slug.

```
account          keyword  (unique, used as _id)
account_display  text
members          nested   { email, name, role: AE|SA|CA|Leader|Other }
notes            text     (free-form account notes)
updated_at       date
updated_by       keyword
```

**Note:** `members` is a nested type and cannot be accessed via ES|QL KEEP. Use `platform.core.get_document_by_id` with `index=account-pursuit-team` and `id=<account>` to retrieve the full member roster.

### `account-rollups`

Computed nightly (+ on every ingest) by the rollup worker. One document per account, `_id` = account name.

```
account              keyword
meeting_count        integer
last_meeting_date    date
first_meeting_date   date
open_action_items    integer
overdue_action_items integer
competitors_seen     keyword[]
sentiment_counts     object  { enthusiastic, positive, neutral, cautious, concerned, skeptical }
latest_sentiment     keyword
tags_frequency       object  (disabled)
meeting_types        keyword[]
authors              keyword[]
momentum_score       float   (higher = better; computed from recency + sentiment + overdue items)
computed_at          date
```

### `action-items`

Denormalized copy of every action item from `granola-meeting-notes`. Updated on ingest. Enables fast "what is overdue?" queries without nested queries.

```
source_note_id  keyword
account         keyword
meeting_date    date
meeting_title   text
description     text
owner           keyword
due_date        date
status          keyword  (open | done)
ingested_at     date
```

### `agent-actions`

Audit log. Every tool call from the agent writes here (fire-and-forget, `refresh: false`).

```
tool_name       keyword
acting_user     keyword
input           object (disabled)
output_summary  text
latency_ms      integer
session_id      keyword
created_at      date
```

### `agent-alerts`

Generated by the nightly alerts worker. Surfaced in the Inbox page.

```
alert_type   keyword  (overdue_action_item | stale_account | at_risk_account)
account      keyword
owner        keyword
severity     keyword  (low | medium | high)
message      text
metadata     object (disabled)
read         boolean
created_at   date
dedup_key    keyword  (prevents duplicate alerts on repeated runs)
```

### `agent-feedback`

Thumbs ratings from the Chat UI (Phase 2) and from `/api/feedback`.

```
session_id   keyword
message_id   keyword
rating       integer  (1 = thumbs up, -1 = thumbs down)
comment      text
tool_calls   object (disabled)
acting_user  keyword
created_at   date
```

### `integrations-slack-users` (Phase 3 scaffold)

```
slack_user_id  keyword
email          keyword
display_name   keyword
```

## 2. Elastic Inference Service (EIS)

Both endpoints are **Elastic-managed** — no API key required, always available on Elastic Serverless.

| Endpoint ID | Task Type | Use |
|---|---|---|
| `.jina-embeddings-v3` | `text_embedding` | Dense embeddings in ingest pipeline + kNN search |
| `.jina-reranker-v2-base-multilingual` | `rerank` | `text_similarity_reranker` in hybrid search |

`npm run setup:elastic` verifies both are available and prints their status.

---

## 3. Ingest Pipeline: `granola-notes-pipeline`

Runs on every document indexed into `granola-meeting-notes`.

| Step | What It Does |
|---|---|
| `set ingested_at` | Timestamp on first ingest only (`override: false`) |
| Auto-tag script | Adds tags based on content keywords (pricing, security, competitive, etc.) |
| Drop blank summary | Prevents inference NPE on empty summaries |
| `.jina-embeddings-v3` on `summary` | Generates `summary_embedding` |
| `.jina-embeddings-v3` on `transcript` | Generates `transcript_semantic` |
| Flag `_has_action_items` | Signals rollup worker to denormalize action items |

---

## 4. Hybrid Search

`ElasticService.hybridSearch()` implements a full RRF + rerank chain with graceful fallbacks:

```
Attempt 1: RRF(BM25, kNN on .jina-embeddings-v3) → text_similarity_reranker(.jina-reranker-v2-base-multilingual)
Attempt 2: RRF(BM25, kNN on .multilingual-e5-small-elasticsearch) → rerank
Attempt 3: BM25 only → rerank
Attempt 4: BM25 multi_match (plain fallback)
```

All attempts use `ignore_unavailable: true` and `allow_partial_search_results: true`.

## 5. Agent: "Account Intelligence Agent"

**Location:** Kibana → Agents → Account Intelligence Agent  
**Endpoint ID:** `account-intelligence-agent`

Created and kept current by `npm run setup:kibana-agent` (idempotent).

### Persona Instructions

The agent has a single instruction set (1,300 words) covering all three personas. The instructions include:
- What indices exist and what their key fields are
- Per-persona behaviour (AE / SA-CA / Leader)
- Tool usage priority and when to use which tool
- Citation format for source notes
- Handling of SFDC stub mode

### ES|QL Tools (10 custom tools, all idempotent via setup script)

| Tool ID | Query Target | Returns |
|---|---|---|
| `aia.get-account-rollup` | `account-rollups` | Rollup stats: meeting count, sentiment, momentum, competitors |
| `aia.get-meeting-timeline` | `granola-meeting-notes` | Chronological meeting list with sentiment + tags |
| `aia.search-notes-by-account` | `granola-meeting-notes` | Full note content: summary, tech env (sub-fields), competitive, budget |
| `aia.list-open-action-items` | `action-items` | Open action items sorted by due date |
| `aia.get-pursuit-team` | `account-pursuit-team` | Account metadata + notes (members via get_document_by_id) |
| `aia.list-all-accounts` | `account-rollups` | All accounts ranked by momentum score |
| `aia.list-alerts` | `agent-alerts` | Unread alerts for an owner |
| `aia.get-commitments` | `granola-meeting-notes` | Notes tagged `has-commitments` with summary (full nested commitments via get_document_by_id) |
| `aia.flag-at-risk-accounts` | `account-rollups` | Accounts with negative/cautious sentiment or momentum < 0 |
| `aia.get-competitors` | `account-rollups` | `competitors_seen` array for an account |

**ES|QL and nested fields:** ES|QL cannot reference `nested` type fields in KEEP clauses. Affected fields:
- `commitments` (nested in granola-meeting-notes) — use `get_document_by_id` for full content
- `action_items` (nested in granola-meeting-notes) — use the `action-items` index instead
- `members` (nested in account-pursuit-team) — use `get_document_by_id` with index=account-pursuit-team

Object fields (not nested) use dot-notation: `technical_environment.current_stack`, `customer_sentiment.overall`, etc.

### Built-in Tools (4 platform tools)

| Tool | Use |
|---|---|
| `platform.core.search` | Free-form natural language search across any index |
| `platform.core.execute_esql` | Ad-hoc ES|QL for questions not covered by named tools |
| `platform.core.get_document_by_id` | Fetch full document including nested fields |
| `platform.core.get_index_mapping` | Inspect field types when writing custom ES|QL |

## 6. Background Workers

### Rollup Worker (`src/server/workers/rollup-worker.ts`)

- **Trigger:** On every successful note ingest (fire-and-forget) + nightly cron (30s after server boot, then every 24h)
- **What it computes:** Meeting count, first/last meeting date, sentiment distribution, competitor set, open/overdue action items, momentum score
- **Momentum score formula:** +1 per meeting in last 30 days, +2 for enthusiastic, +1 for positive, -1 for concerned/skeptical, -0.5 per overdue item
- **Also:** Denormalizes action items from the ingested note into the `action-items` index

### Alerts Worker (`src/server/workers/alerts-worker.ts`)

- **Trigger:** 60s after server boot, then every 24h
- **Checks:** Overdue action items (high severity), stale accounts (no meeting in 30+ days, medium severity), at-risk accounts (concerned or skeptical sentiment, high/medium severity)
- **Dedup:** Alert is not created if `dedup_key` already exists in `agent-alerts`

## 9. Salesforce Integration (Stub Mode)

Three SFDC tool stubs are wired into the MCP server:
- `sfdc_update_opportunity` — queues for manual Salesforce entry
- `sfdc_log_call` — same
- `sfdc_create_task` — same

All calls: (1) return `{ success: true, stub: true, id: "00Q..." }`, (2) write an audit entry to `agent-actions`.

To go live: set `SALESFORCE_MODE=live` in `.env`. The factory `createSalesforceService()` returns `LiveSalesforceService` which can be filled with real API calls. Tool signatures, audit log, and agent prompts are identical — only the implementation changes.

## 10. Eval Harness

Gold Q&A sets: `docs/eval/gold-ae.jsonl`, `gold-sa-ca.jsonl`, `gold-leader.jsonl`  
5 questions × 3 personas = 15 gold questions (extend to ~30 per persona before Phase 2 launch).

Run: `npm run run:eval`

Metrics captured: latency p50/p95, source coverage (did the right indices appear in the answer), accuracy (stub — connect `AGENT_BUILDER_URL` for live eval).

## 11. Slack Scaffold (Phase 3)

Files: `src/server/integrations/slack/`
- `handler.ts` — slash command handler, no live posting

## 12. Running Commands

```bash
npm run setup:elastic        # Idempotent: create indices, verify EIS endpoints, update pipeline
npm run setup:kibana-agent   # Idempotent: create/update Kibana agent + 10 ES|QL tools
npm run seed:lookups         # Seed reference data (accounts, tags, meeting types)
npm run dev                  # Start both servers (frontend: 5173, backend: 3001)
npm run run:rollups          # Manually trigger rollup computation for all accounts
npm run run:alerts           # Manually trigger alerts worker
npm run run:eval             # Run eval harness against gold Q&A sets
```
